## Agents

### Agents window (Preview)

The [Agents window](https://aka.ms/VSCode/Agents/docs) is a dedicated companion window optimized for exploring, iterating on, and reviewing agent sessions across projects and machines. This release focuses on making it faster to move between sessions and on preserving your context across reloads.

#### Navigate between sessions

As you accumulate more sessions, jumping to the one you need becomes the main bottleneck. This release adds several ways to move through your sessions without leaving the keyboard.

* **Sessions Picker**: Press `kbstyle(Ctrl+R)` (`kbstyle(Cmd+R)` on macOS) to open a quick pick with two groups: **recently opened** and **other sessions**. The active session is pre-selected, and you can search across both the session title and its folder. Press `kbstyle(Enter)` to open the selected session, `kbstyle(Cmd/Ctrl+Enter)` to open it to the side, or `kbstyle(Right Arrow)` to open it in the background while keeping the picker open.
* **Go back and forward**: Press `kbstyle(Ctrl+Tab)` to go back and `kbstyle(Ctrl+Shift+Tab)` to go forward through the sessions you have visited, in most-recently-visited order.
* **Previous and Next session**: The **Go to Previous Session** and **Go to Next Session** commands step through the visible sessions list in display order, respecting grouping, filtering, and collapsed sections, and clamp at the list edges. Use `kbstyle(Alt+Up)` / `kbstyle(Alt+Down)` (or `kbstyle(Ctrl+PageUp)` / `kbstyle(Ctrl+PageDown)`; `kbstyle(Cmd+Alt+Left)` / `kbstyle(Cmd+Alt+Right)` on macOS).
* **Focus a session by position**: Press `kbstyle(Ctrl+1)` through `kbstyle(Ctrl+9)` (`kbstyle(Cmd+1)` to `kbstyle(Cmd+9)` on macOS) to focus the Nth visible session in the grid, left to right.

#### Restore sessions on reload

Reloading or reopening the Agents window no longer loses your layout. The previously visible sessions and their state are now restored automatically, so you land back where you left off. This includes:

* The visible sessions grid: which sessions were open, their left-to-right order, the active session, and any sticky/pinned sessions.
* Per-session layout, including auxiliary bar visibility, the active view container, and the editors open in each session.
* Sessions list state, including the pinned and read status of each session.

#### Background send for new sessions

Previously, starting a new session meant waiting for it to load before you could compose the next one. You can now fire off a request in the background: press `kbstyle(Alt+Enter)` (or hold `kbstyle(Alt)` and select **Send**) in the new session view. The view resets immediately and retains its state, such as the selected model and context, clearing only the query text, so you can keep queuing up requests. Each started session appears in the sessions list once it commits. Sending with `kbstyle(Enter)` still navigates into the new session as before.

#### Close all sessions

The new **Close All Sessions** command removes every visible session in a single pass and returns you to the new-session view. Trigger it with `kbstyle(Ctrl+K Ctrl+W)` (`kbstyle(Cmd+K Cmd+W)` on macOS) while a session has focus; the command is also always available in the Command Palette.

#### Single-file diff in the Changes view (Preview)

**Setting**: `setting(sessions.changes.openSingleFileDiff)`

By default, selecting a file in the **Changes** view opens the multi-file diff editor. Enable `setting(sessions.changes.openSingleFileDiff)` to instead open a focused single-file diff editor when you click an added, modified, or deleted file, so you can concentrate on one change at a time. When the setting is enabled, the redundant **Open Changes** inline action is hidden.

#### Widen the editor with the side bar chevron

When you open a file inside the Agents window, a chevron toggle now appears in the editor title bar, to the right of the tabs. Select it to collapse the secondary side bar (auxiliary bar) and widen the editor, and select it again to bring the side bar back. The chevron direction reflects the current side bar visibility.

## Editor Experience

### Integrated Browser

#### History

**Setting**: `setting(workbench.browser.maxHistoryEntries)`

The integrated browser now remembers a history of visited pages. History items are surfaced as suggestions when typing in the URL bar, and can be managed using `kb(workbench.action.browser.showHistory)` from within a browser tab. The maximum number of history items to remember can be adjusted via `setting(workbench.browser.maxHistoryEntries)`.

![The URL bar in the integrated browser, with an input of "Wiki" showing several Wikipedia pages from history](images/1_124/browser-history.png)

#### Improved toolbar customizability

Previously, "Add Element to Chat" and "Toggle Developer Tools" were the only actions that could be made persistently visible in the right side of the browser toolbar.

Now, all actions that appear within the overflow menu can also be surfaced peristently by right-clicking on the toolbar area to the right of the URL input:
![The context menu allowing the user to toggle visible menu items](images/1_124/browser-toolbar.png)

#### Faster agentic text entry

Previously, typing text and Pressing `Enter` required two separate tool calls. Now, the `typeInPage` tool supports a `submit` parameter to allow agents to type text and press Enter in one tool call.

This is a small optimization that makes a big difference in common text entry scenarios.

![A tool call showing the agent typed text into a page and pressed enter](images/1_124/browser-type-tool.png)

## Enterprise

### Enterprise-managed Copilot plugin policies (Experimental)

VS Code now reads policy from the same configuration file that already drives [enterprise plugin standards for Copilot CLI](https://docs.github.com/en/copilot/how-tos/administer-copilot/manage-for-enterprise/manage-agents/configure-enterprise-plugin-standards), so a single policy definition applies to both clients. In the future, VS Code and the CLI will align further on policy management from this shared source.

Enterprise admins can now centrally control which chat plugins and plugin marketplaces are available to their developers, instead of asking each developer to configure them locally.

Introduced in 1.123, three new policy-backed settings can be configured through the Copilot enterprise settings file or via existing MDM solutions:

* `setting(chat.plugins.enabledPlugins)`: an allowlist of plugin IDs that the organization explicitly enables or disables.
* `setting(chat.plugins.extraMarketplaces)`: additional plugin marketplaces the organization wants to make available.
* `setting(chat.plugins.strictMarketplaces)`: when enabled, only marketplaces supplied by policy are trusted.

Plugins blocked by policy remain visible but appear disabled, and marketplaces managed by policy are tagged as such in the marketplace picker.
