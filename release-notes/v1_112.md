# Visual Studio Code 1.112

_Release date: March 18, 2026_

<!-- DOWNLOAD_LINKS_PLACEHOLDER -->

---

Welcome to the 1.112 release of Visual Studio Code. This release includes various **improvements across agent management and the developer experience**.

* [**Copilot CLI**](#copilot-cli): steer and queue messages in CLI sessions, and preview changes before delegating to Copilot CLI.

* [**Integrated browser**](#integrated-browser): debug web apps, use context menus, and zoom in the built-in browser.

* [**Agent plugins**](#agent-plugins): manage plugins per workspace and install individual plugins from a repository.

* [**MCP auto-allow configuration**](#mcp-auto-allow-configuration): mark MCP servers and tools as enabled or auto-allowed directly in `mcp.json`.

* [**Chat customizations discovery**](#chat-customizations-discovery): discover customization files from parent repositories and configure hooks at the user level.

* [**Chat UX improvements**](#chat-ux-improvements): browse image carousels in responses, let agents read images from disk, and export agent debug logs.

Happy Coding!

---

VS Code is rolling out gradually to all users. Use **Check for Updates** in VS Code to get the latest version immediately.

To try new features as soon as possible, [**download the nightly Insiders build**](https://code.visualstudio.com/insiders), which includes the latest updates as soon as they are available.

## Agents

### Copilot CLI

**Message steering and queueing:**
In this release we added steering and queueing of messages to Copilot CLI sessions.
Learn more about [message steering and queueing](https://code.visualstudio.com/docs/copilot/chat/chat-sessions#_send-messages-while-a-request-is-running) in the documentation.

![Steering and queueing.](images/1_112/copilot_cli_steering.png)

**View changes before delegating:**
In this release we added the ability for one to see the changes that can be migraded over to the worktree created when delegating to Copilot CLI, allowing one to make a more informed descision.

![Screenshot showing the changes.](images/1_112/copilot_cli_delegation_view_changes.png)

### Terminal file link support for Copilot CLI session paths

The terminal's file link detection now recognizes paths generated by Copilot CLI that reference files under the `~/.copilot/session-state/` directory. Previously, these paths were not resolved correctly because the built-in link detector did not know about the Copilot CLI session-state directory structure. Both absolute and relative paths are now handled: absolute and tilde-prefixed paths open directly, while relative paths are resolved against the active session-state directory with a fallback to workspace folders.

This feature is enabled by default and can be toggled with the `setting(github.copilot.chat.cli.terminalLinks.enabled)` setting.

<video src="images/1_112/cli-terminal-links.mp4" title="Video showing clickable file links in Copilot CLI terminal output opening files in the editor." autoplay loop controls muted></video>

### Improved Toolbar and Bypass Approvals in CLI

Last iteration, we added a secondary toolbar and a new permissions picker in the local chat. This iteration, we bring those changes to the CLI as well - you can select either `Default Permissions` or `Bypass Approvals`. Similar to Local chat, `Autopilot`  is still only enabled by default in Insiders, but configurable using the `setting(chat.autopilot.enabled)` setting.

Learn more about [Autopilot and Agent Permissions here](https://code.visualstudio.com/updates/v1_111#_autopilot-and-agent-permissions)!


## Chat

### Troubleshoot agent debug logs with /troubleshoot (Preview)

A new `/troubleshoot` skill helps you investigate unexpected chat agent behavior by analyzing agent debug logs directly in the conversation. When your chat customizations, such as instructions, skills, or agents, are not loading or behaving as expected, when a chat request is unexpectedly slow, or when you suspect network issues, you can ask the agent to troubleshoot the issue.

The `/troubleshoot` skill reads the JSONL debug log files that are exported from the chat session and provides insights into why tools or subagents were used or skipped, why instructions or skills did not load, what contributed to slow response times, and whether network connectivity problems occurred.

To use the `/troubleshoot` skill, enable both of the following settings:

* `setting(github.copilot.chat.agentDebugLog.enabled)` - enables agent debug logging
* `setting(github.copilot.chat.agentDebugLog.fileLogging.enabled)` - writes debug logs to JSONL files on disk

After enabling these settings, reload VS Code for the changes to take effect. Then, ask the agent to analyze your chat session with the `/troubleshoot` skill.

### Agent Debug Log Export and Import (Preview)
To view agent debug logs panel, the following settings need to be enabled
<img width="1657" height="121" alt="image" src="https://github.com/user-attachments/assets/ced69b26-c6f9-46c1-9785-f20761933c21" />
For more information on Agent Debug Log panel, see [Agent Debug Logs](https://code.visualstudio.com/docs/copilot/chat/chat-debug-view)

From the debug panel, you can now export a debug session log to Open Telemetry JSON (OTLP format) file to share it with others or analyze it offline. You can also import a previously exported file to view it in the Agent Debug Logs panel.

To export a session:

* Open the Agent Debug Logs panel and navigate to the session you want to export.

* Select the Export icon (download) in the top-right toolbar of the panel.

* Choose a location to save the JSON file.

If no session is selected, VS Code shows a notification that there is no active debug session to export.

To import a session:

* Select the Import icon (upload) in the top-right toolbar of the Agent Debug Logs panel.

* Select a previously exported JSON file.

The imported session opens in the Agent Debug Logs panel with its overview and metrics, just like a live session.

> [!NOTE]
>Importing files larger than 50 MB shows a warning dialog with the actual file size. Consider trimming the file or exporting a shorter session if you encounter this warning.


## MCP

### Sandbox locally running MCP servers

You can now run locally configured stdio MCP servers in a sandboxed environment on macOS and Linux. Sandboxed servers have restricted file system and network access, which helps reduce the risk of running local servers with broader access than they need.

To enable sandboxing, set `"sandboxEnabled": true` for a server in your `mcp.json` file. When a sandboxed server needs access to an additional folder or domain, VS Code prompts you to grant that permission and updates the sandbox configuration for that `mcp.json` file. All servers defined in the same `mcp.json` file share that sandbox configuration.

Sandboxing for locally running MCP servers is not currently available on Windows. Remote scenarios, such as WSL and SSH, continue to work.

## Editor Experience

### Auto-close Find dialog after search

A new `setting(editor.find.closeOnResult)` setting lets you automatically close the Find widget after a match is found and move focus back to the editor.

The setting is `false` by default, preserving the existing behavior where the Find dialog stays open after a search.

### Integrated Browser

We are continuing to improve the Integrated Browser in VS Code.

Noteworthy changes in this release include support for debugging, context menus, and independent zoom.

#### Debug Integrated Browsers

A new `editor-browser` debug type enables debugging of integrated browser tabs with both Launch and Attach configurations.

Most options from existing `msedge` and `chrome` configurations are supported, so migration is often as simple as changing the type of an existing configuration in `launch.json`:
<video src="images/1_112/browser-debugging.mp4" title="Video showing debugging in the integrated browser." autoplay loop controls muted></video>


#### Context menus

Right clicking in a browser page now exposes common options such as copy/paste, open in new tab, and inspect.
![An open context menu in the integrated browser](images/1_112/browser-context-menu.png)


#### Independent browser zoom

The integrated browser now supports its own zoom, independent from the VS Code window's zoom level.

You can zoom using the **Zoom In**, **Zoom Out**, and **Reset Zoom** actions in the menu to the right of the URL bar, or with the following keyboard shortcuts when the integrated browser has focus:

* **Zoom in**: `kb(workbench.action.browser.zoomIn)`
* **Zoom out**: `kb(workbench.action.browser.zoomOut)`
* **Reset zoom**: `kb(workbench.action.browser.resetZoom)`

As in a standard browser, integrated browser remembers the zoom level per website. That is, if you zoom in or out on one page of a site, all other pages on that site will use that same zoom level.

You can also configure a default zoom level with the `setting(workbench.browser.pageZoom)` setting. If that setting is not set or is set to "Match Window", the default zoom level will match the VS Code window's zoom level.

## Terminal

### Improved IME composition for the terminal

When typing with an Input Method Editor (IME) near the right edge of the terminal, the composition preview text could previously overflow outside the terminal boundaries. The composition view is now constrained to the available space between the cursor and the terminal's right edge. As new characters are entered, older characters are progressively hidden, keeping the preview text within the terminal viewport. This matches the behavior of other modern terminals such as Ghostty.
