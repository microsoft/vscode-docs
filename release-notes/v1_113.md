## Agents

### MCP Support in Copilot CLI & Claude agents

This release we have added support for MCP Servers in Copilot CLI & Claude agents.
* [VS Code MCP documentation](https://code.visualstudio.com/docs/copilot/customization/mcp-servers)
* [MCP server development guide](https://code.visualstudio.com/docs/copilot/guides/mcp-developer-guide)

This means that MCP Servers you have registered in VS Code will be bridged to Copilot CLI and Claude agents.

### Forking Sessions in Copilot CLI (Experimental)

**Setting**: `setting(github.copilot.chat.cli.forkSessions.enabled)`

This release we have added the ability to fork sessions in Copilot CLI.
Learn more about [forking a chat session](https://code.visualstudio.com/docs/copilot/chat/chat-sessions#_fork-a-chat-session) in the documentation.
To enable this feature, turn on the `setting(github.copilot.chat.cli.forkSessions.enabled)` setting.
![Screenshot showing forking of a Copilot CLI session.](images/1_113/copilot-cli-fork-sessions.png)

### Forking Sessions in Claude agent

This release we have also added the ability to fork sessions in Claude agent.
Learn more about [forking a chat session](https://code.visualstudio.com/docs/copilot/chat/chat-sessions#_fork-a-chat-session) in the documentation.

### Claude agent now powered by official SDK APIs to list sessions

The Claude agent SDK has given us an official API to list out sessions and their messages and we've now adopted that. Previously, we relied on parsing Claude JSONL files on disk which had a risk of being out of sync if Claude changed their structure. If you experienced issues with the Claude agent not showing all your sessions or messages, this should now be resolved.

### Configurable thinking effort in model picker

Models that support reasoning, such as Claude Sonnet 4.6 and GPT-5.4, now show a **Thinking Effort** submenu directly in the model picker. You can use this to control how much reasoning the model applies to each request without navigating to VS Code settings.

Select a reasoning model in the picker and click the arrow to reveal the available effort levels. The selected effort level is displayed in the picker label, for example "GPT-5.3-Codex · Medium". The available effort levels vary by model. Non-reasoning models do not show the submenu.

![Screenshot showing the model picker with a Thinking Effort submenu displaying Low, Medium, and High effort levels for a reasoning model.](images/1_113/chat-model-picker.png)

Your selected effort level is persisted per model and is retained across conversations.


## Chat Experience

### Images Preview for Chat Attachments

**Setting**: `setting(imageCarousel.chat.enabled)`

When working with images in chat whether you attached screenshots to your request or the agent generated images via tool calls, you can now click any image attachment to open it in a full **Images Preview**.

The viewer opens as a modal overlay and supports:

- **Navigation**: Browse all images from the current chat session using arrow buttons, keyboard arrows, or the thumbnail strip at the bottom.
- **Sections**: Images are grouped by conversation turn, so you can see which images came from a particular request or response.
- **Zoom & pan**: Click to zoom in, use Option+Click (Mac) or Ctrl+Click (Windows/Linux) to zoom out, or scroll/pinch to zoom continuously. At high zoom levels, scroll to pan around the image.

<video src="images/1_113/image-carousel.mp4" title="Video showing the image carousel feature in chat." autoplay loop controls muted></video>

In addition, you can also right-click an image file or folder in the Explorer and select **Open in Images Preview** to browse images in the same viewer

### Use self-signed certificates in the integrated browser

When developing web applications that depend on secure HTTPS connections, it's usually necessary to use a self-signed certificate during testing.

Under normal circumstances, such certificates shouldn't be trusted. Previously, any site that presented an untrusted certificate would simply fail to load in the integrated browser, and there was no option to bypass.

Now, similar to most browsers, you can choose to temporarily trust a certificate that can't be verified, to unblock development in these scenarios:

![The integrated browser displaying a certificate error with options to go back or proceed](images/1_113/browser-cert-error.png)

Proceeding will allow connections to the current host using that certificate for one week. The URL bar will show that the connection isn't secure, with the option to revoke trust:

![A popup in the URL bar showing the current page is not secure, with an option to "Revoke and Close"](images/1_113/browser-trust-dialog.png)


### Improved browser tab management

Managing open tabs can already be difficult. As we encourage more use of integrated browser tabs, we are also adding more controls to easily manage them.

#### Command: `Quick Open Browser Tab...`

This command opens a quick pick displaying all open browser tabs and allowing them to be quickly filtered, focused, and closed.

<video src="images/1_113/browser-quick-open.mp4" title="Video showing the quick open feature switching between browser tabs." autoplay loop controls muted></video>

This can be triggered by pressing `Ctrl/Cmd+Shift+A` while a browser is currently focused, or via a new shortcut button in the VS Code title bar, visible when a browser tab is open:

![The VS Code title bar, showing a new button with a globe icon](images/1_113/browser-title-bar-button.png)

The visibility of this button is configurable via the `setting(workbench.browser.showInTitleBar)` setting.

#### Command: `Close All Browser Tabs`

Browser tab context menus now have an option to close all browser tabs in the same group, similar to the existing "Close All" item:

![A context menu on an open integrated browser tab, showing an item "Close All Browser Tabs"](images/1_113/browser-close-all-item.png)

Browser tabs across all groups can also be closed via the command palette.
