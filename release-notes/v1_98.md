### Copilot Edits

#### Refined editor integration

We have polished the integration of Copilot Edits with code and notebook editors:

* We don't scroll along anymore as changes are being made. The viewport remains in place, making it easier to focus on what changes.
* We have renamed the edit review actions from "Accept" to "Keep" and "Discard" to "Undo". This reflects better what's happening: changes for Copilot Edits are live, they are applied and saved as they are made and users keep or undo them.
* Lastly, when you have decided to keep or undo a file, the next file is automatically revealed.

The recording below shows how edits are applied and saved as they are made. The live preview updates and the user decided to "Keep" the changes. Undoing and further tweaking is also possible.

<video src="images/1_98/edits_editor.mp4" title="Changes from Copilot Edits are saved automatically and the user decided to keep them" autoplay loop controls muted></video>

#### Refreshed UI

In preparation for unifying Copilot Edits with Copilot Chat, we've given Copilot Edits a facelift. Files that are attached and not yet sent, are now rendered as regular chat attachments. Only files that have been modified with AI are added to the changed files list, which is located above the chat input part. Related file suggestions are rendered below chat attachments when configuring `setting(chat.renderRelatedFiles)`.

### Removed working set size limits

There is no longer a limit of 10 files for the working set in Copilot Edits. Additionally, we've removed the client-side rate limit of 14 interactions per 10 minutes. Note that service-side usage rate limits still apply.

### Smoother permission upgrade flows

For Copilot users who host their repositories on GitHub, you're able to leverage several features including advanced code searching, the `@github` chat participant, and more! For private GitHub repositories, however, we need to have permission to interact with your repositories on GitHub. For a while, this was presented with our usual VS Code Authentication flow with a modal dialog showing up when you invoke certain functionality (asking `@workspace` or `@github` a question, using the `#codebase` tool, etc.).

To smooth this out we've introduced this confirmation in chat:

![a confirmation dialog](images/1_98/confirmation-auth-dialog.png)

Not only is it not as jarring as a modal, but it has new functionality:

1. **Grant:** You're taken through the authentication flow as you've seen it before (via the modal).
2. **Not Now:** Your choice will be remembered, and we won't bother you again until your next VS Code window session. The only exception to this is if the feature needs this additional permission to function, like `@github`.
3. **Never Ask Again:** Your choice will be remembered and persisted via the setting `setting(github.copilot.advanced.authPermissions)`. Any feature that needs this additional permission will fail.

It's important to note that this confirmation does not confirm/deny Copilot, the platform, access to your repositories. This is only how VS Code's Copilot experience authenticates. To configure what Copilot can access, please read the docs [on content exclusion](https://docs.github.com/en/copilot/managing-copilot/configuring-and-auditing-content-exclusion/excluding-content-from-github-copilot).

### Attachable Problems

You now can attach problems to your chat request, either by dragging them from the Problems view into the chat view, by typing `#problems` in chat, or by clicking the paperclip üìé button. You can attach specific problems, all problems in a file, or all files in your codebase.

### Copilot Vision (Public Preview)

We're introducing end-to-end vision support in this version of GitHub Copilot Chat. This lets you attach images and interact with images in Copilot Chat prompts. For example, if you're encountering an error while debugging, quickly attach a screenshot of VS Code and ask Copilot to help you resolve the issue.

![Screenshot that shows an attached image in a Copilot Chat prompt. Hovering over the image shows a preview of it.](images/1_97/image-attachments.gif)

You can now attach images using several methods:

- Drag and drop images from your OS or from the Explorer view
- Paste an image from your clipboard
- Attach a screenshot of the VS Code window (click the paperclip üìé button > Screenshot Window)

A warning is shown if the selected model currently does not have the capability to handle the file type. `GPT 4o`, `Claude 3.5 Sonnet`, and `Gemini 2.0 Flash` will support image attachments. Currently, the supported image types are `JPEG/JPG`, `PNG`, `GIF`, and `WEBP`.

## Accessibility

### Copilot Edits

We made Copilot Edits much more accessible. There are now audio signals for files with modifications and for changed regions (insertions, modifications, and deletions). The accessible diff viewer is now available for modified files. Just as in diff editors select `kb(chatEditor.action.showAccessibleDiffView)` to enable it.

### `activeEditorState` window title variable

We have a new `setting(window.title)` variable, `activeEditorState`, to indicate editor information such as modified state, the number of problems, and when a file has pending Copilot Edits to screen reader users. When in Screen Reader Optimized mode, this is appended by default and can be disabled with `setting(accessibility.windowTitleOptimized:false)`.

## Source Control

### Updated View Titles

The addition of the "Source Control Graph" view to the "Source Control" viewlet has highlighted the duplication present in the source control view titles: "Source Control Repositories", "Source Control", and "Source Control Graph". This milestone we have revisited the titles of the views so that they are shorter and do not duplicate the viewlet title: "Repositories", "Changes", and "Graph".

### Discard Untracked Changes Improvements

Over the years we have received multiple reports about data loss as discarding an untracked file would permanently delete the file even though VS Code shows a modal dialog making it clear that the file will be permanently deleted. Starting this milestone discarding an untracked file will move the file to the Recycle Bin/Trash when possible so that the file can be easily recovered. You can disable this functionality using the setting(git.discardUntrackedChangesToTrash) setting.

### Diagnostics Commit Hook (Experimental)

This milestone we have added a new commit hook that will prompt the user if there are any unresolved diagnostics for the changed files. This is currently an experimental feature that can be enabled using the setting(git.diagnosticsCommitHook.Enabled:true) setting. By default, the commit hook will prompt for any error level diagnostics but the diagnostics sources and levels can be customized using the setting(git.diagnosticsCommitHook.Sources) setting. Please give it a try and let us know your feedback.

## Terminal

### Terminal completions improvements (Preview)

We've significantly improved shell completions across Bash, Zsh, Fish, and PowerShell by adding completion specs (`git` for example), refining command-line parsing for better suggestions, and enhancing file and folder completions. Enable this feature with `setting(terminal.integrated.suggest.enabled:true)`.

### Improved parsing

We now leverage Fig‚Äôs parsing to support subcommands and arguments across a wider range of commands.

### Configurable completion behavior

Gain precise control over when completions appear using `setting(terminal.integrated.suggest.quickSuggestions)`.

### Inline suggestion detection

With enhanced ghost text detection, inline suggestions now appear as the top completion result and can be accepted with `Tab`. Configure this behavior with `setting(terminal.integrated.suggest.inlineSuggestion)`.

### Enhanced documentation support

Built-in commands for Zsh and Bash now include descriptions and documentation, providing details on available arguments. This information is sourced directly from each shell‚Äôs help documentation.

### Improved sorting

Commands with completion specs and built-in commands now appear before those fetched from `$PATH` as executables.

### Alias support

Command aliases are now displayed with a distinct icon, with the corresponding command shown as a description.

### New icons

Enhanced parsing introduces specific icons for different completion types, improving clarity and usability.

### `$CDPATH` support

`$CDPATH` is a shell feature that enables seamless navigation to directories by treating specified paths as implicit search locations. With `$CDPATH` set, users can `cd` into a directory without needing to specify its full path.

For example, setting `CDPATH=/dev` allows running `cd vscode` to navigate directly to `/dev/vscode`, regardless of the current working directory.

This functionality is now integrated into the suggest widget, enhancing usability by recognizing and completing paths based on `$CDPATH`.

## Extension Authoring

### Bring authentication interactive settings of `forceNewSession` to `createIfNone` and rename `AuthenticationForceNewSessionOptions` to `AuthenticationGetSessionPresentationOptions`

> [!IMPORTANT]
> We are renaming `AuthenticationForceNewSessionOptions` to `AuthenticationGetSessionPresentationOptions` and leaving a deprecated `AuthenticationForceNewSessionOptions` for now. There is no functional difference so this is not a breaking change in the runtime, but you should update your code to use `AuthenticationGetSessionPresentationOptions` instead of `AuthenticationForceNewSessionOptions` since it will be removed in the future.

Looking at these two authentication calls:
```ts
vscode.authentication.getSession(provider, scopes, { createIfNone: options });
vscode.authentication.getSession(provider, scopes, { forceNewSession: options });
```

`createIfNone` and `forceNewSession` will now take in either a `boolean` or a `AuthenticationGetSessionPresentationOptions`:
```ts
/**
 * Optional options to be used when calling {@link authentication.getSession} with interactive options `forceNewSession` & `createIfNone`.
 */
export interface AuthenticationGetSessionPresentationOptions {
	/**
	 * An optional message that will be displayed to the user when we ask to re-authenticate. Providing additional context
	 * as to why you are asking a user to re-authenticate can help increase the odds that they will accept.
	 */
	detail?: string;
}
```
_[full typings can be found here](https://github.com/microsoft/vscode/blob/release/1.98/src/vscode-dts/vscode.d.ts#L17520-L17551)_...

This is a new addition for `createInNone`, but it's a modification for `forceNewSession` which use to take in a `AuthenticationForceNewSessionOptions` which had the same signature of this new `AuthenticationGetSessionPresentationOptions`.

If you are explicitly using `AuthenticationForceNewSessionOptions`, you will see it is marked as deprecated and you should replace it with `AuthenticationGetSessionPresentationOptions` as `AuthenticationForceNewSessionOptions` will be removed in a future version.

It's important to note that the only thing that is changing here are the types. **There is no runtime change so this is not a breaking change from that perspective.**

Additionally, the `authLearnMore` [proposed API](https://github.com/microsoft/vscode/blob/release/1.98/src/vscode-dts/vscode.proposed.authLearnMore.d.ts), has been updated from `AuthenticationForceNewSessionOptions` to `AuthenticationGetSessionPresentationOptions`.

### Refined Snippet API

You can now control the whitespace normalization when inserting snippets. This applies to the [`insertSnippet`](https://github.com/microsoft/vscode/blob/c202fb0bcfc7ac863f90756bdf668e801b96901d/src/vscode-dts/vscode.d.ts#L1306)-API and to the [`SnippetTextEdit`](https://github.com/microsoft/vscode/blob/c202fb0bcfc7ac863f90756bdf668e801b96901d/src/vscode-dts/vscode.d.ts#L3753)-API and control is the indentation of additional lines of snippets are adjusted or not

```js
const snippet = `This is an indented
    snippet`;

// keepWhitespace: false, undefined
function indentedFunctionWithSnippet() {
    return `This is an indented
        snippet`; // adjusted indentation
}

// keepWhitespace: true
function indentedFunctionWithSnippet() {
    return `This is an indented
    snippet`; // original indentation
}

```

## Engineering


### Dev-time tracking of leaked disposables

VS Code uses the disposable-pattern for explicit resource management, e.g. to close files, clean up DOM elements, or remove event listeners. Not disposing of resources means memory is wasted and memory usage accumulates over time. We are constantly on the hunt for such leaks and have now added another tool for this. We utilize the [`FinalizationRegistry`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/FinalizationRegistry)-API as it will inform us when an object has been garbage collected. If such an object represented a `disposable`, which has not been disposed of, then we have a leak. These are collected and shown to the developers of VS Code so that we can clean things up as we go.
