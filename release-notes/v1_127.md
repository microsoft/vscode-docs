## Agents

### Agents window (Preview)

The [Agents window](https://aka.ms/VSCode/Agents/docs) is a dedicated companion window optimized for exploring, iterating on, and reviewing agent sessions across projects and machines. This release brings new ways to organize the sessions list so you can keep a busy list of sessions manageable.

#### Session list grouping

When you run several agent sessions at once, the sessions list can grow quickly and become hard to scan. You can now organize the sessions list into groups to keep related sessions together. Create your own custom groups, and collapse group headers to tidy up the list and focus on what matters.

Each group also offers quick actions: you can start a new session directly in a group, or mark all of its sessions as done in one step.

<video src="images/1_127/sessions-group.mp4" title="The video shows how to group sessions in the Agents window. The user creates a new group and drags sessions into it." autoplay controls muted></video>

#### Drag and drop in the sessions list

The sessions list now supports drag and drop, so you can arrange it exactly how you want. Reorder sessions by dragging them up or down, and drag session group and workspace headers to rearrange the list.

Drag and drop also makes organizing sessions easier:

* Drag a session onto a group to add it to that group.
* Drop a session onto the **Pinned** section to pin it.
* Select multiple sessions to move them together as a block.

As you drag, an insertion line shows where the items will land.

<video src="images/1_127/sessions-dnd.mp4" title="The video shows how to drag and drop sessions in the Agents window." autoplay controls muted></video>

#### Stay on top of your agent sessions with chat input banners

When a coding agent session has an open pull request, the Agents window now surfaces banners directly above the chat input, so you get a nudge to act on failing checks and incoming feedback right where you're working. Each banner gives you one click to either fix the issue or go see it, without leaving your conversation:

* **CI failures:** When checks on the pull request fail, a banner shows how many checks failed (for example, "2 of 5 checks failed") with quick actions to **Fix Checks** to kick off a fix in the agent, or **Reveal Checks** to jump straight to the failing checks in the Changes view.

<video src="images/1_127/input-banner-ci.mp4" title="A banner shows above the input with the text '8 out of 22 checks failed, 7 pending'. The user presses reveal checks which shows the pull request checks in the secondary side bar. The user then presses fix checks on the input banner so the agent addresses the failing checks." autoplay controls muted></video>

* **Pull request comments:** When new review comments come in, a banner shows the comment count with actions to **Address Comments** to hand them to the agent to act on, or **Reveal Comments** to open them in the editor.

<video src="images/1_127/input-banner-pr.mp4" title="A banner shows above the input with the text 2 pr comments. The user presses reveal comments which shows the comments in the editor. The user then presses address comments on the input banner so the agent addresses them." autoplay controls muted></video>

#### Onboarding tours (Experimental)

Getting started with agents can be daunting if you're not sure what they can do for you. Onboarding tours are now available in the Agents window to help you get up to speed quickly. These guided walkthroughs highlight key capabilities and show you how to make the most of working with agents, so you can discover the best ways to delegate tasks and stay productive from day one.

We're experimenting with these tours to find the most helpful way to introduce new users to the experience.

<video src="images/1_127/onboarding.mp4" title="The video shows an onboarding example. The user sends a message. A Pulse animation shows on the new session button. When the user presses it the workspace picker and the isolation picker are spotlighted and explained in the new session view." autoplay controls muted></video>

#### Editor gutter feedback when reviewing agent changes

When reviewing an agent's changes, pointing it at the exact code you want changed should be effortless. You can now leave feedback directly from the editor gutter: hovering over a line reveals an **add feedback** glyph in the gutter, and selecting it drops a comment on that line, making it quicker to direct an agent to a specific spot in the code.

This release also brings a round of polish to the agent feedback experience, with refinements to the feedback input, hover behavior, and overall visual consistency.

<video src="images/1_127/agent-feedback-add-action.mp4" title="The video shows how to add feedback to an agent's changes. The user hovers over a line, clicks the add feedback glyph, and types a comment." autoplay controls muted></video>

### Troubleshoot agent behavior with /troubleshoot

The troubleshoot skill (invoked via `/troubleshoot` command) helps with diagnosing chat issues by analyzing chat session logs and surfacing insights into the agent's behavior. For example, to investigate why custom instructions were ignored or responses are slow.

With this release, you can now use troubleshoot command to diagnose agent host sessions (local and remote sessions). In the Agents window, type `/troubleshoot` in the chat input, followed by a `#session`, select the session you want to troubleshoot, followed by a question or description of the issue you're experiencing.

## Chat

### Subagent credits

When an agent delegates work to a subagent, it can be difficult to know the cost of the delegated work. You can now hover over a subagent section in the chat response to see the credits used by that subagent for additional transparency.

![image of subagent cost hover](images/1_127/subagent_cost_hover.png)

### Built-in Ollama provider deprecation

Ollama now maintains an [official VS Code extension](https://marketplace.visualstudio.com/publishers/Ollama), which is the recommended way to use local Ollama models in chat. Because the extension is owned by the Ollama team, you get faster support for new models and capabilities than the built-in provider could offer.

As a result, the built-in Ollama provider is now deprecated. If you currently use it to run local models with [Bring Your Own Key (BYOK)](https://code.visualstudio.com/docs/copilot/customization/language-models#_bring-your-own-language-model-key), install the official extension and remove the built-in provider to continue using your Ollama models without interruption. The following video shows how to remove the deprecated provider.

### Integrated Browser

#### Camera, Location, Devices, and more

The integrated browser now supports per-site permissions. This enables pages to use many more web APIs including:

- Geolocation
- Camera / Microphone
- Sensors (accelerometer, etc.)
- Clipboard
- Devices (Bluetooth / USB / Serial / HID)

When a page requests a permission, you will be prompted to allow or deny the request -- just as you would expect in a traditional browser.

<img src="images/1_127/browser-permission-prompt.png" alt="A system dialog showing 'app.zoom.us wants access to Microphone' with options to Allow, Block, or Cancel." width="282" height="221" />

Permissions can be easily managed for the current site via the "Site Permissions" browser menu item.

#### Agent Tools are now GA

Browser tools let an agent open pages in the integrated browser, read content and console errors, take screenshots, and click, type, and navigate to verify its own work — all without an external MCP server. After several milestones in preview, browser tools are now generally available and enabled by default with the `setting(workbench.browser.enableChatTools)` setting.

A big thank you to everyone who ran the preview, filed issues, and shared feedback. Your testing directly shaped the per-session tab isolation, the explicit page-sharing controls, and the permission model that ship in this release.

Ask an agent to build and validate a web app, or follow the step-by-step [Build and test web apps with browser agent tools](https://code.visualstudio.com/docs/agents/guides/browser-agent-testing-guide) guide to see the closed build-test-fix loop in action. For the full reference, see [Browser tools for agents](https://code.visualstudio.com/docs/debugtest/integrated-browser#_browser-tools-for-agents).

Administrators can govern browser tools through enterprise policy: disable them entirely with the `BrowserChatTools` policy, or restrict which domains agent tools can reach with agent network filtering (`ChatAgentNetworkFilter` plus allow and deny domain lists). See [Configure AI settings for your organization](https://code.visualstudio.com/docs/enterprise/ai-settings#_configure-agent-network-filtering).
